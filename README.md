# dooservice agent

Daemon that runs on each VPS to manage Odoo instances via Docker. Receives jobs from the orchestrator over NATS JetStream, executes them, and streams real-time progress back. Distributed as a single compiled binary — no Python runtime required on the VPS.

| Repo | Description |
|---|---|
| [`dooservice/agent`](https://github.com/dooservice/agent) | **This repo** — VPS agent daemon |
| [`dooservice/orchestrator`](https://github.com/dooservice/orchestrator) | Central API server |
| [`dooservice/core`](https://github.com/dooservice/core) | Shared SDK, models, transport, protocol |

## Platform overview

```
Dashboard (React)
    └── REST / WebSocket ──► Orchestrator (FastAPI)
                                └── NATS JetStream ──► Agent (per VPS)
                                                        ├── Docker Engine
                                                        ├── Postgres + PgDog
                                                        ├── Traefik (reverse proxy)
                                                        └── Backups (local / S3)
```

Each agent manages a single VPS. It subscribes to its own NATS subject plus a regional subject, executes jobs dispatched by the orchestrator, and sends a heartbeat every 30 seconds with CPU, RAM, and disk metrics.

## Install on a VPS

```bash
curl -fsSL https://raw.githubusercontent.com/dooservice/agent/main/install.sh | sudo bash
```

**Requirements:** Ubuntu 22.04+, root access, internet access.

The installer will:

1. Install Docker from the official Ubuntu repository
2. Download the `dooservice-agent` binary from GitHub Releases to `/usr/local/bin/`
3. Create system user `dooservice` (uid `1500`) and add to the `docker` group
4. Detect server specs (RAM, CPU cores) and calculate optimal Postgres/PgDog settings
5. Prompt for NATS credentials, base domain, TLS mode, DNS provider, and S3 (optional)
6. Generate `/var/lib/dooservice/agent.toml` with `chmod 600`
7. Install and enable the `dooservice-agent` systemd service
8. Run `bootstrap configure` and `bootstrap start` (Postgres → PgDog → Traefik)

## Configuration

Config is loaded from `/var/lib/dooservice/agent.toml` (created by the installer). See [`agent.example.toml`](agent.example.toml) for all options with inline comments.

Config is resolved in this order: `--config` flag → `DOOSERVICE_AGENT_CONFIG` env var → `/var/lib/dooservice/agent.toml`.

```toml
nats_url               = "nats://nats.example.com:4222"
nats_user              = "agent"
nats_password          = "secret"
region                 = "us-east"
heartbeat_interval     = 30    # seconds
max_concurrent_backups = 3

[sdk]
data_dir = "/var/lib/dooservice"   # stores agent.db, pgdog/, letsencrypt/, projects/
debug    = false

[sdk.postgres]
superuser_password   = "<auto-generated by installer>"
max_connections      = 200     # kept low — PgDog handles per-tenant connection pooling
shared_buffers       = "16GB"  # ~25% of RAM
effective_cache_size = "48GB"  # ~75% of RAM

[sdk.pgdog]
workers           = 16          # match CPU core count
pooler_mode       = "session"   # required for Odoo (LISTEN/NOTIFY and long polling)
default_pool_size = 12

[sdk.proxy]
base_domain      = "clientes.example.com"
server_ip        = "1.2.3.4"
acme_email       = "admin@example.com"
dns_provider     = "cloudflare_token"
secondary_domains = []   # additional root domains served by this agent

[sdk.dns]
cloudflare_api_token = "..."

[sdk.s3]          # optional — omit to keep backups on local disk only
endpoint   = "https://s3.amazonaws.com"
access_key = "..."
secret_key = "..."
bucket     = "dooservice-backups"
```

## CLI reference

```bash
# ── Daemon ────────────────────────────────────────────────────────────────────
dooservice-agent serve [--config PATH]
# Connects to NATS, subscribes to job subjects, starts the backup scheduler.

# ── Infrastructure stack ──────────────────────────────────────────────────────
dooservice-agent bootstrap configure     # set base domain, TLS, DNS provider
dooservice-agent bootstrap start         # bring up Postgres → PgDog → Traefik
dooservice-agent bootstrap stop
dooservice-agent bootstrap rebuild
dooservice-agent bootstrap status        # show running state of all 3 services

# ── Traefik (reverse proxy) ───────────────────────────────────────────────────
dooservice-agent proxy show              # show current proxy config
dooservice-agent proxy start|stop|rebuild|status

# ── Postgres ──────────────────────────────────────────────────────────────────
dooservice-agent pg start|stop|rebuild|status

# ── PgDog (connection pooler) ─────────────────────────────────────────────────
dooservice-agent pgdog start|stop|rebuild|status

# ── systemd service ───────────────────────────────────────────────────────────
dooservice-agent agent install           # write unit file + daemon-reload
dooservice-agent agent start|stop|restart|enable|disable
dooservice-agent agent status
dooservice-agent agent logs [--no-follow] [--lines 200]
```

## Job handlers

The agent handles the following job kinds dispatched by the orchestrator:

| Kind | Description |
|---|---|
| `env.provision` | Provision a new Odoo environment (Docker container + DB + proxy route) |
| `env.clone` | Clone an existing environment to a new branch/instance |
| `env.start` | Start a stopped environment container |
| `env.stop` | Stop a running environment container |
| `env.rebuild` | Pull the latest Odoo image and recreate the container |
| `env.delete` | Remove environment container, volume, and proxy route |
| `env.logs` | Stream container logs (request-reply) |
| `env.update_workers` | Reconfigure Odoo worker count |
| `env.update_backup_policy` | Enable or disable automatic daily backups |
| `backup.create` | Create a full or partial database backup |
| `backup.restore` | Restore a backup to an environment |
| `backup.multipart.complete` | Register and finalize an externally uploaded backup |
| `backup.drop` | Mark a backup as dropped (keeps the DB record) |
| `git.pull` | Pull latest commits for an environment's branch (autodeploy) |
| `domain.set` | Assign a custom domain + request TLS certificate |
| `domain.remove` | Remove a custom domain and its proxy route |
| `domain.verify` | Verify DNS propagation for a custom domain |
| `project.delete` | Remove all resources for a project |
| `query.environments` | Return all environments for a project (request-reply) |
| `query.backups` | Return all backups for an environment (request-reply) |
| `query.backup.download_url` | Return a presigned S3 download URL (request-reply) |
| `query.backup.multipart_url` | Initiate S3 multipart upload, return URLs (request-reply) |

Progress frames emitted during long-running jobs:

```json
{ "type": "progress",  "job_id": "...", "stage": "pulling image", "pct": 40 }
{ "type": "completed", "job_id": "...", "result": { ... } }
{ "type": "failed",    "job_id": "...", "error": "..." }
```

## Backup scheduler

The daemon runs a built-in APScheduler that fires daily at **03:00 local time** for each production environment with `auto_backup_enabled = true`.

Guarantees:

- Only **production** environments are scheduled (development environments are excluded)
- **One backup per environment** at a time — if a manual backup is in progress, the scheduled run is skipped
- At most `max_concurrent_backups` (default `3`) running across all environments simultaneously
- The previous day's backup file (S3 object or local file) is **deleted before** uploading the new one — storage stays bounded even if the new backup fails mid-upload
- The database record is **never deleted** — previous backups are marked `DROPPED` for permanent audit history
- Graceful shutdown: waits up to 120 seconds for in-flight backups before releasing database connections
- Invalid timezone strings fall back to UTC with a warning instead of crashing the daemon
